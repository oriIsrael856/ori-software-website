# ארכיטקטורה — Ori Software Solutions

מסמך זה מתעד את ההחלטות הארכיטקטוניות של הפרויקט בפורמט **ADR
(Architecture Decision Record)** — הפרקטיקה המקובלת בתעשייה לתיעוד "למה",
לא רק "מה". כל רשומה מציגה הקשר, החלטה, וטרייד-אופים במודע.

> העיקרון המנחה: **ארכיטקטורה חכמה היא לא הארכיטקטורה עם הכי הרבה שכבות —
> היא זו שמתאימה לבעיה, ויודעת להסביר את עצמה.** להוסיף מורכבות זה קל;
> להצדיק כל שכבה זה הקשה.

---

## ADR-001: אתר סטטי מלא (SSG) — אפס שרתים בזמן ריצה

**הקשר**: אתר תדמית + תוכן יומי. תעבורה לא צפויה (קמפיין לינקדאין יכול להביא
פיק פתאומי). תקציב: מינימלי.

**החלטה**: כל עמוד מרונדר מראש ב-build (`○ Static` / `● SSG` בפלט ה-build).
אין רנדור בזמן בקשה, אין דאטאבייס חי, אין state בשרת.

**למה זה עומד בסקייל**: עמוד סטטי מוגש מ-CDN גלובלי. מיליון גולשים בו-זמנית
הם בעיה של ה-CDN (שנבנה בדיוק לזה), לא שלנו. אין שרת שקורס, אין connection
pool שנגמר, אין cold start.

**בונוס אבטחה**: מה שלא רץ — לא נפרץ. אין endpoint דינמי = אין SQL injection,
אין SSRF, שטח תקיפה מזערי.

**טרייד-אוף במודע**: עדכון תוכן דורש build (כ-דקה). בתדירות של חידוש אחד ביום —
זול לחלוטין. אם נגיע לתוכן בתדירות גבוהה: ISR (Incremental Static Regeneration)
הוא שדרוג של שורה אחת, בלי לשנות ארכיטקטורה.

## ADR-002: תוכן כקוד (Content-as-Code) — Markdown בגיט, לא CMS

**הקשר**: פיצ'ר "חידוש יומי" צריך אחסון תוכן. האופציות: DB (Postgres),
Headless CMS (Sanity/Contentful), או קבצים בריפו.

**החלטה**: קבצי Markdown עם frontmatter בתיקיית `content/til/`.

**נימוקים**:
1. **גרסאות חינם** — כל שינוי תוכן הוא commit עם היסטוריה מלאה ו-revert בקליק.
2. **אפס תלות חיצונית** — אין ספק צד ג' שמעלה מחירים, נופל, או נסגר.
3. **התוכן עובר את אותו pipeline כמו הקוד** — code review, ולידציה ב-build (ADR-005), פריסה אטומית.
4. **אוטומציה** — סוכן AI יכול לכתוב קובץ ולפתוח commit; אין API keys לנהל.

**מתי נחליף**: אם יהיו כותבים לא-טכניים, או תוכן שנוצר מהאתר עצמו (תגובות,
לידים) — אז DB. לא לפני שיש צורך מוכח. ראו "עיקרון YAGNI" בהמשך.

## ADR-003: שכבת גישה לתוכן — Repository Pattern

**הקשר**: קוד שקורא קבצים היה יכול להתפזר בכל עמוד שצריך תוכן.

**החלטה**: מודול יחיד — [lib/til.ts](lib/til.ts) — הוא הדרך היחידה לגשת לתוכן.
העמודים מקבלים אובייקטים מטופסים (`TilPost`), לא נתיבי קבצים.

**מה זה קונה**: אם מחר התוכן עובר ל-DB או ל-CMS (ADR-002), משתנה קובץ אחד.
העמודים לא יודעים ולא ישתנו. זו ההגדרה של coupling נמוך.

**פרט מימוש**: `cache()` של React עוטף את הקריאה — הדיסק נקרא פעם אחת גם אם
חמישה עמודים מבקשים את הרשימה באותו build (memoization לכל רנדור).

## ADR-004: קונפיגורציה במקור אמת יחיד — lib/site.ts

כל פרט עסקי (שם, וואטסאפ, לינקים, מיצוב) חי בקובץ אחד בלבד. אף מחרוזת עסקית
לא מקודדת בקומפוננטות. שינוי מספר טלפון = שורה אחת, בלי חיפוש-החלפה בכל הריפו.
דפוס: Single Source of Truth.

## ADR-005: ולידציה ב-Build Time — Fail Fast

**הקשר**: קובץ תוכן עם frontmatter שבור (תאריך בפורמט שגוי, כותרת חסרה) היה
מרנדר עמוד פגום בפרודקשן בלי שאף אחד ישים לב.

**החלטה**: סכמת `zod` מאמתת כל קובץ תוכן בזמן ה-build. קובץ פסול = ה-build
נכשל עם הודעה שמצביעה על הקובץ והשדה המדויקים.

**העיקרון**: להעביר כשלים שמאל (shift left) — הכי זול לתפוס שגיאה בזמן פיתוח,
יקר יותר ב-build, והכי יקר בפרודקשן מול לקוח. באתר סטטי, ה-build הוא שער
האיכות האחרון — אז הוא קשוח.

## ADR-006: אבטחה בשכבת ה-HTTP — כותרות קשוחות כברירת מחדל

**הקשר**: גם אתר סטטי חשוף ל-clickjacking, MIME sniffing, והזרקות אם
תוכן זדוני נכנס אי-פעם.

**החלטה** ([next.config.ts](next.config.ts)):

| כותרת | תפקיד |
|---|---|
| `Content-Security-Policy` | רק משאבים מהדומיין שלנו; חוסם הזרקת סקריפטים חיצוניים, `frame-ancestors 'none'` נגד clickjacking |
| `Strict-Transport-Security` | HTTPS בלבד לשנתיים, כולל סאב-דומיינים |
| `X-Content-Type-Options: nosniff` | הדפדפן לא "מנחש" סוגי קבצים |
| `Referrer-Policy` | לא מדליפים URL מלא לאתרים חיצוניים |
| `Permissions-Policy` | מצלמה/מיקרופון/מיקום כבויים — לא צריכים, אז חסומים |
| `poweredByHeader: false` | לא מפרסמים גרסת פריימוורק לסורקי פגיעויות |

**טרייד-אוף מתועד**: `script-src` כולל `'unsafe-inline'` כי סקריפטי ההידרציה
של Next בעמודים סטטיים הם inline. האלטרנטיבה — CSP מבוסס nonce — דורשת רנדור
דינמי לכל בקשה ומוותרת על כל היתרונות של ADR-001. בחרנו במודע: אתר סטטי בלי
קלט משתמשים, וקטור ההזרקה כמעט לא קיים; היום שנוסיף טופס עם עיבוד — נשקול מחדש.

**עקרון ההרשאה המינימלית** חוזר בכל שכבה: ה-CSP מתיר רק מה שנחוץ,
ה-Permissions-Policy חוסם יכולות שלא בשימוש, ואין אף secret בצד לקוח.

## ADR-007: גבול שרת/לקוח — RSC כברירת מחדל, אינטראקטיביות בעלים בלבד

**הקשר**: ב-App Router כל קומפוננטה היא Server Component אלא אם הוכרז אחרת.
הפיתוי: לשים `"use client"` בכל מקום ולגמור עניין.

**החלטה**: `"use client"` רק בעלים (leaves) שבאמת מחזיקים state או מאזינים
לאירועים: סינון תגיות, כפתורי שיתוף, אנימציות. כל היתר — כולל פרסור Markdown
ועיצובו — רץ בשרת בזמן build ולא שולח JavaScript לדפדפן.

**למה**: כל KB של JS בצד לקוח נמדד בזמן טעינה בסלולר. הגבול הזה הוא
החלטת ביצועים, לא סגנון.

## ADR-008: עיצוב כטוקנים — CSS Custom Properties כשכבת תיווך

כל צבע מוגדר פעם אחת כמשתנה סמנטי (`--accent`, `--surface`, `--line`)
ב-[globals.css](app/globals.css). כשהמותג התחלף מליים לכחול — הוחלפו ערכים
בקובץ אחד. הקומפוננטות מדברות בשמות תפקידים ("accent"), לא בערכי צבע.
זה גם התשתית המוכנה ל-dark/light mode עתידי: החלפת ערכי משתנים, אפס שינוי קומפוננטות.

## מפת דפוסי עיצוב (Design Patterns) בפרויקט

| דפוס | איפה | מה הוא פותר |
|---|---|---|
| Repository | `lib/til.ts` | ניתוק העמודים ממנגנון האחסון |
| Single Source of Truth | `lib/site.ts`, טוקני CSS | שינוי במקום אחד מתפשט לכל האתר |
| Composition (HOC-style wrapper) | `components/Reveal.tsx` | אנימציית כניסה אחידה לכל אלמנט בלי לשכפל לוגיקה |
| Fail Fast | ולידציית zod ב-build | שגיאות תוכן נתפסות לפני פרודקשן |
| Registry (מתוכנן) | `components/showcase/registry.ts` | הוספת קומפוננטה למעבדה בלי לגעת בעמודים |
| Facade | `whatsappLink` ב-`lib/site.ts` | הרכבת ה-URL במקום אחד; הצרכנים לא מכירים את פורמט wa.me |
| Memoization | `cache()` ב-`lib/til.ts` | קריאת דיסק אחת לרנדור, לא אחת לקומפוננטה |

## עיקרון YAGNI — מה החלטנו *לא* לבנות (וזו החלטה ארכיטקטונית)

- **אין דאטאבייס** — אין נתונים דינמיים. יתווסף כשיהיו (טפסים/לידים/מונים).
- **אין state manager** (Redux/Zustand) — ה-state היחיד הוא תגית פעילה בפילטר. `useState` מספיק.
- **אין microservices/containers** — מונולית סטטי על CDN מנצח כל תשתית מנוהלת-עצמית בסקייל הזה, גם בעלות וגם באמינות.
- **אין ספריית UI חיצונית** — Tailwind + קומפוננטות משלנו. פחות תלויות = פחות עדכוני אבטחה של אחרים.

כל "אין" כאן הוא לא חוסר — הוא החלטה עם תנאי כניסה מוגדר לשינוי.

## מפת דרכים אבטחתית — מה נדרש כשהאתר יגדל

| טריגר | צעד אבטחה נדרש |
|---|---|
| טופס יצירת קשר עם API route | ולידציית קלט בשרת (zod), rate limiting, honeypot נגד בוטים, סניטיזציה לפני שליחת מייל |
| מונה העתקות במעבדה (KV) | rate limiting לפי IP, אימות מקור בקשה |
| תגובות/תוכן גולשים | סניטיזציה של HTML, moderation, ו-CSP קשוח יותר (הסרת unsafe-inline) |
| כל secret ראשון | משתני סביבה ב-Vercel בלבד, אף פעם בריפו; הפרדת סביבות dev/prod |
| תלויות | `npm audit` בכל CI run; עדכוני dependabot |
